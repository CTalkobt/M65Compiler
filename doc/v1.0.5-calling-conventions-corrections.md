## Key Corrections

### 1. Frame Pointer Initialization: UNCONDITIONAL in v1.0.4

**Old Documentation (Misleading):**
> "Frame pointer setup is currently unconditional (planned optimization: lazy initialization only when needed)"

**Corrected Documentation:**
> "The frame pointer ($FD/$FE) is unconditionally set up at function entry for all stack-convention functions, even if the function doesn't use frame-relative variables. Lazy initialization is planned for v1.1 as an optimization."

**Why This Matters:**
- Users need to know that frame pointer setup is a **required overhead** in v1.0.4
- Leaf functions cannot avoid FP setup (11 bytes + 20 cycles)
- v1.1 will optimize this away, but v1.0.4 does not

**Code Behavior:**
```asm
; v1.0.4: ALWAYS executed at function entry
tsx
txa
clc
adc #1
sta $FD
lda #$01
adc #0
sta $FE
.frameptr_zp $FD
```

### 2. Frame Pointer Stability After Function Calls

**Old Documentation (Incorrect):**
> "After each function call (JSR), the frame pointer must be recalculated because stack pointer may have changed."

**Corrected Documentation:**
> "The frame pointer is set once at function entry and remains stable throughout the function. Frame-relative offsets are computed at compile time and are only valid relative to the initial FP. The FP is never recalculated during function execution, even after JSR calls."

**Why This Was Wrong:**
- Previous investigation identified that FP recalculation after JSR was a **bug**, not a feature
- Recalculating FP broke frame-relative offsets (they became invalid)
- Fix: Frame pointer remains stable throughout execution

**Impact:**
- All frame-relative addresses remain valid throughout function
- Multiple JSR calls within a function work correctly
- Struct parameters access still works after calling other functions

### 3. Frame Layout Clarity

**Old Documentation (Vague):**
```
[Caller's return address (2 bytes)]
[Stack parameters (N bytes)]
← SP before JSR
← FP after setup (SP + 1, handling page crossing)
[Frame variables (M bytes)]
[Vregs (K bytes)]
← SP during function execution
```

**Corrected Documentation (Precise):**
```
STACK MEMORY (Low to High Address):
[Return address: 2 bytes]
FP = SP+1 ← Frame pointer points here
[Frame parameters: N bytes]
[Frame locals: M bytes]
[Frame vregs: K bytes]
← SP during function execution
```

**Key Clarifications:**
1. **FP = SP + 1** — Frame pointer points to the first allocated frame byte
2. **Offsets from FP** — All variables accessed via `FP + offset`
3. **Fixed at entry** — Frame layout is static, not dynamic

### 4. Return Value Handling: AXYZ Registers

**Old Documentation (Incomplete):**
Limited to conceptual description without addressing Z register complexity.

**Corrected Documentation (Complete):**
```
| Type | Byte 0 | Byte 1 | Byte 2 | Byte 3 |
|------|--------|--------|--------|--------|
| char | A | — | — | — |
| int/short/ptr | A (lo) | X (hi) | — | — |
| long | A | X | Y | Z |
```

**Clarification on 32-bit returns:**
- All four registers (AXYZ) are used for 32-bit values
- Compiler uses `lda.fp` / `ldx.fp` / `ldy.fp` / `ldz.fp` or equivalent sequences
- Special care needed when Z must be loaded: may use `taz` to move A→Z then reload A

### 5. Struct Return Mechanism: Hidden Pointer

**Old Documentation (Present but Underspecified):**
> "Structs are NOT returned in registers. Instead: [description of hidden parameter]"

**Corrected Documentation (Complete with Examples):**

**Mechanism:**
1. Caller allocates static `__return_struct_temp`
2. Caller passes temp address as **hidden first parameter** (before regular parameters)
3. Callee writes struct to temp address
4. Caller copies temp to destination

**Assembly Example:**
```asm
leax.fp @__return_struct_temp    ; Get temp address
phx                              ; Push hi-byte (hidden param)
pha                              ; Push lo-byte (hidden param)
lda #2                           ; Regular params pushed after
ldx #0
phx
pha
lda #1
ldx #0
phx
pha
jsr struct_returning_function
; After JSR, struct is in static temp location
```

**Why Static Temp?**
- Avoids frame pointer corruption during struct writes
- Predictable location for caller to access result
- Works with any struct size
- Simple compiler implementation

### 6. Page Boundary Crossing

**Old Documentation (Mentioned but Not Explained):**
> "Potential issue in frame pointer arithmetic... needs verification"

**Corrected Documentation (Fully Explained):**

**How Frame Pointer Handles Page Boundaries:**
```asm
tsx                ; X ← SPL
txa                ; A ← SPL
clc
adc #1             ; A ← SPL + 1 (may produce $00 with carry)
sta $FD            ; FP_lo ← SPL + 1
lda #$01           ; Page $01 for main stack
adc #0             ; Add carry if SPL+1 wrapped
sta $FE            ; FP_hi ← $01 + (carry ? 1 : 0)
```

**Example:**
- SPL = $FF
- SPL + 1 = $100 (wraps to $00 with carry)
- FP_lo = $00, FP_hi = $01 + carry = $02
- Result: FP = $0200 (correct page boundary handling)

### 7. ZP Calling Convention: Fixed Parameters

**Old Documentation (Correct):**
Mostly accurate, but emphasized limitations.

**Corrected Documentation (Unchanged):**
ZP convention documentation was already accurate.

**Confirmed Details:**
- Parameters in $20-$21, $22-$23, $24-$25, $26-$27
- No frame pointer setup
- No stack operations
- 4-parameter maximum
- Clobber tracking in object files

### 8. Varargs: Stack-Only Feature

**Old Documentation (Correct):**
Varargs support already clearly documented as stack-convention-only.

**Corrected Documentation (Unchanged):**
Varargs section remains accurate.

**Confirmed Behavior:**
- `va_start(args, lastparam)` computes address past last fixed parameter
- `va_arg(args, type)` reads N bytes (N = sizeof(type)) and advances
- Only works with stack convention
- ZP convention cannot support varargs

## Optimization Roadmap (v1.1+)

Based on corrected understanding:

### v1.1 Priority 1: Lazy Frame Pointer Initialization
- **Cost:** 11 bytes + 20 cycles per leaf function
- **Detection:** Scan for frame-relative variable access
- **Implementation:** Conditional FP setup at entry based on analysis
- **Benefit:** ~10-15% code size reduction for typical programs

### v1.1 Priority 2: Skip FP Recalc for Zero-Argument Calls
- **Cost:** ~10 cycles per zero-argument call
- **Implementation:** Check `argBytes > 0` before FP recalc
- **Benefit:** 20+ cycles saved in recursive functions and callbacks

### v1.1 Priority 3: Leaf Function Detection
- **Cost:** 20+ cycles for any call within leaf function
- **Implementation:** Build call graph, mark leaf functions
- **Benefit:** Eliminate FP recalc overhead entirely for leaf functions
