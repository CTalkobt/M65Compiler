# ln45 — Linker for .o45 Object Files

`ln45` links multiple `.o45` relocatable object files into a single flat binary or PRG executable. It resolves symbol references, merges segments, and applies relocations.

## Usage

```bash
ln45 [options] <file1.o45> [file2.o45 ...]
```

## Options

| Flag | Description |
|------|-------------|
| `-o <file>` | Output filename (default: `a.out`, or `a.prg` if `-prg`) |
| `-prg` | Output as PRG with 2-byte load address header (auto-detected from `.prg` extension) |
| `-basic` | Output as PRG with BASIC `SYS` stub at `$0801` (code starts at `$080D`; auto-run on MEGA65/C64) |
| `-t <addr>` | Set text segment base address (hex, default: `2000`) |
| `-d <addr>` | Set data segment base address (hex, default: after text) |
| `-b <addr>` | Set BSS segment base address (hex, default: after data) |
| `-z <addr>` | Set zero page base address (hex, default: `02`) |
| `-L <dir>` | Add a library search directory (also see `CC45_LIB` environment variable) |
| `-l <lib>` | Link against a `.lib` archive (selective linking — only needed members are pulled in) |
| `-D<sym>=<val>` | Define a linker symbol (hex value). Overrides weak definitions. Example: `-D__heap_end=$C000` |
| `-m` | Print linker map (symbol addresses) after linking |
| `-M <file>` | Write detailed linker map file (segment layout, per-object contributions, symbols) |
| `--no-thunks` | Error on calling convention mismatches instead of generating thunks |
| `-Wthunk` | Warn on convention mismatches (default behavior — thunks are always warned) |
| `-V` / `--version` | Display version and exit |
| `-?` | Display help |

## Environment Variables

| Variable | Description |
|----------|-------------|
| `CC45_LIB` | Colon-separated list of directories to search for library files specified with `-l`. If a library path is not found as given, each directory in `CC45_LIB` is checked (followed by any `-L` directories). |

## Linking Process

The linker performs three passes:

### 1. Layout
Segments from all input objects are concatenated in input order:
- All text segments are merged into one contiguous text block
- All data segments are merged into one contiguous data block
- BSS and ZP lengths are summed
- Each object gets an offset within the merged segment

### 2. Symbol Resolution
- **Exports** from all objects are collected into a global symbol map
- Each export's final address is computed: `segment_base + object_offset + symbol_offset`
- **Imports** are verified — every imported symbol must have a matching export
- Duplicate exports are rejected with an error

### 3. Relocation
- Each object's relocation table is decoded
- For external references (`SEG_EXTERNAL`), the import name is looked up in the global symbol map
- For internal cross-segment references, the target segment's final base + object offset is computed
- The patched address is written into the merged segment body

## Relocation Types

| Type | Patch | Description |
|------|-------|-------------|
| `R_WORD` | 2 bytes | 16-bit absolute address |
| `R_LOW` | 1 byte | Low byte of 16-bit address |
| `R_HIGH` | 1 byte | High byte of 16-bit address |
| `R_LINEAR24` | 3 bytes | 24-bit linear address (MEGA65) |
| `R_LINEAR32` | 4 bytes | 32-bit linear address |
| `R_SEGADR` | 3 bytes | Segment address (bank:addr) |

## Examples

### Basic linking

```bash
# Compile two C files to objects
cc45 -c main.c -o main.o45
cc45 -c math.c -o math.o45

# Link into a PRG
ln45 -prg -o program.prg main.o45 math.o45
```

### Auto-running BASIC PRG (recommended for MEGA65/C64)

```bash
cc45 -c main.c -o main.o45

# Link with BASIC SYS stub — auto-runs on load
ln45 -basic -o program.prg main.o45 -l c45.lib
```

This produces a PRG that loads at `$0801` with a `10 SYS 2061` BASIC stub. Code starts at `$080D`. The program auto-runs when loaded via `LOAD"PROGRAM",8,1` then `RUN` (or auto-start).

### Assembly object linking

```bash
# Assemble to objects
ca45 -c main.s -o main.o45
ca45 -c utils.s -o utils.o45

# Link with explicit text base at $0800
ln45 -t 0800 -o program.bin main.o45 utils.o45
```

### Inspect before linking

```bash
# Check symbols
nm45 main.o45 utils.o45

# Link with map output
ln45 -prg -m -o program.prg main.o45 utils.o45
```

Example map output:
```
Symbol map:
  00002000 _main
  0000203c _add
  00002068 _mul2
  0000208b _result
```

### Detailed linker map file

```bash
ln45 -prg -M program.map -o program.prg main.o45 crt0.o45 puts.o45
```

Example map file (`program.map`):

### Custom memory layout

```bash
ln45 -t 0800 -d 4000 -b 6000 -z 02 -o rom.bin boot.o45 kernel.o45
```

## Calling Convention Thunks

When linking objects compiled with different calling conventions (stack vs ZP), the linker detects mismatches via per-function `.func_flags` metadata in `.o45` objects and generates bridge thunks automatically.

**Behavior:**
- By default, a warning is emitted for each mismatch and a thunk is generated
- `--no-thunks` turns mismatches into hard errors (no thunk generation)
- `-Wthunk` is equivalent to the default (warn + generate)

**Thunk types:**
- **Stack→ZP (`__thunk_s2z_`)**: Copies stack parameters to ZP param slots, then `JMP` to callee
- **ZP→Stack (`__thunk_z2s_`)**: Pushes ZP params onto stack, `JSR` callee, cleans stack, `RTS`

Call sites targeting mismatched functions are automatically redirected to the appropriate thunk. Thunks add runtime overhead (~20-30 cycles per call), so matching conventions at compile time is preferred.

```bash
# Default: warn on mismatches, generate thunks
ln45 -basic -o program.prg main.o45 -l c45.lib

# Error on mismatches (no thunks)
ln45 --no-thunks -basic -o program.prg main.o45 -l c45.lib
```

## Error Messages

| Error | Cause |
|-------|-------|
| `undefined symbol '_foo'` | An object imports `_foo` but no object exports it |
| `duplicate symbol '_foo'` | Two objects both export `_foo` |
| `relocation offset out of range` | A relocation entry points past the end of a segment |
| `invalid symbol index` | A relocation references a non-existent import table entry |

## Output Format

Without `-prg`: raw flat binary (text + data, gap-filled if needed).

With `-prg`: 2-byte little-endian load address header followed by the binary. The load address is the text base (`-t` value). This format is directly loadable on the MEGA65/C64.

BSS is not included in the output — it must be zeroed at runtime (by `crt0` or the startup code).

## Function Metadata (ZP Calling Convention)

When object files are compiled with `-fzpcall` or contain `__fastcall__` functions, each function may include metadata directives that describe its zero-page and register usage. These are stored as 16-byte function attribute records in the `.o45` symbol table.

| Directive | Purpose |
|-----------|---------|
| `.zp_uses $03, $04, ...` | ZP slots read as input parameters |
| `.zp_clobbers $03, $04, ...` | ZP slots this function may overwrite |
| `.zp_release $07, $08, ...` | ZP slots consumed (caller need not restore) |
| `.reg_clobbers A, X, Y` | CPU registers not preserved across the call |
| `.flag_clobbers C, N, Z` | Processor flags not preserved |

The linker reads this metadata to build call graphs and compute transitive clobber sets for interprocedural optimization. Objects without function attributes are treated conservatively as "clobbers all, releases none".

## Diagnostics

The linker emits warnings to stderr for potential issues:

| Warning | Meaning |
|---------|---------|
| `warning: symbols 'X' and 'Y' overlap at $NNNN (different segments)` | Two symbols from different segment types (e.g., TEXT and DATA) were placed at the same address. This usually indicates a bug in the assembler's segment assignment. |
| `ln45: warning: unused global symbol 'X' (defined in FILE.o45)` | A symbol was marked as `.global` (exported) but is not referenced by any other module in the program. Internal symbols (names starting with `__`) are suppressed. This helps identify dead code. |

## Limitations

- **Custom segments**: The `.o45` format supports four segment types: TEXT, DATA, BSS, and ZP. User-defined segments (`.segment "myname"`) are mapped to TEXT in `.o45` output. Custom segments work correctly in flat binary mode (`ca45` without `-c`) but lose their identity in relocatable objects. Use the standard segment directives (`.code`, `.data`, `.bss`, `.zp`) for relocatable builds.
