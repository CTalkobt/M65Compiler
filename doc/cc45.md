# cc45 — Mega65 C Compiler

`cc45` is a C compiler designed for the MEGA65 (45GS02). It translates a subset of the C language into optimized assembly code.

## Features
- **Function Declarations**: Full support for function definitions with parameters.
- **Local Variables**: Support for local variables allocated on the stack.
- **Structures & Unions**: Full support for `struct` and `union` definitions, members, and dot/arrow operators.
- **Anonymous Aggregates**: Support for nested structures and unions without names, allowing direct member access from the parent scope.
- **Control Flow**: `if`/`else` statements, `while` loops, and `for` loops.
- **Increment/Decrement**: Optimized `++` and `--` (prefix and postfix) using simulated opcodes.
- **Nested Expressions**: Robust handling of complex arithmetic expressions and nested function calls.
- **String Pooling**: Automatic identification and pooling of string literals into a data section using `.text` (PETSCII). Character literals (`'A'`, `'d'`) are also PETSCII-encoded, so comparisons between characters from strings and char literals work correctly.
- **Procedure System**: Translates C functions into `PROC` blocks for `ca45`, leveraging high-level procedure syntax.
- **Volatile Support**: Full support for the `volatile` keyword to prevent optimization of hardware-mapped or interrupt-modified variables.
- **Static Assertions**: Support for `_Static_assert` to perform compile-time validation of constant expressions.
- **Function Specifiers**: Support for `_Noreturn` to indicate that a function does not return to its caller, enabling optimizations to skip return instructions.
- **Alignment**: Support for C11 `_Alignas` and `_Alignof` to manage data alignment for global variables, local variables, and struct members.
- **Generic Selections**: Support for C11 `_Generic` expressions for compile-time type-based dispatch.
- **Designated Initializers**: Support for C99 designated initializers (`{.x=1, .y=2}` for structs and `{[2]=3}` for arrays) to initialize specific struct members and array elements.
- **Flexible Array Members**: Support for C99 flexible array members in structs (e.g., `struct s { int type; char data[]; };`) for variable-length data structures.
- **Compound Literals**: Support for C99 compound literals (e.g., `(int[3]){1, 2, 3}` or `(struct point){.x=10, .y=20}`) to create unnamed aggregate values.
- **Dead Code Elimination**: Optimization to remove unreachable code following infinite loops (e.g., statements after `while(1)` without `break`).
- **Weak Symbols**: `#pragma weak` marks the next function or global variable as a weak export. Weak definitions can be overridden by strong definitions from other translation units at link time, enabling optional feature hooks and default implementations.
- **CRT Pragmas**: `#pragma crt <option>` controls C runtime startup behavior:
    - `no_0100_stack` — Relocatable stack base (`__sp_base`) for non-page-$01 stacks.
    - `no_bssinit` — Skip BSS zeroing at startup. By default, uninitialized globals are zeroed by `_init_bss`.
    - `no_zp_save` — Disable automatic Zero Page preservation ($08-$FF). Useful for standalone firmware where ZP state is not shared with an OS.
    - `exit_rts` — (Default) Return to caller (BASIC SYS, monitor). Saves/restores the caller's full 16-bit SP. Provides `__exit` label for `exit()`.
    - `exit_halt` — Infinite loop after `main` returns.
    - `exit_brk` — Break to debugger after `main` returns.
    - `heap` — Enable dynamic memory allocation (`malloc`, `free`, `calloc`, `realloc`). Links `crt_heap.o45` and calls `_init_heap_crt` during startup.
- **Arrays**: Native support for one-dimensional and multi-dimensional arrays (`type name[rows][cols]`) with subscript indexing, automatic pointer decay, and row-major layout. Multi-dimensional arrays compute correct strides at each indexing level (e.g., `int a[3][4]` uses stride 8 for the row index and stride 2 for the column index). Struct arrays (`struct point pts[10]`) are supported with element-indexed member access (`pts[i].x`). `sizeof` returns the full array size. Braced initializer lists are supported: `int arr[5] = {1, 2, 3};` initializes elements in order, zero-filling any remaining elements. Empty initializers `= {}` zero-fill the entire array. Works for both global arrays (emits `.byte`/`.word` data directives) and local arrays (pushes elements onto the stack). Trailing commas are allowed.
- **Ternary Operator**: Support for conditional expressions (`cond ? then : else`).
- **Bitwise Operators**: Verified support for `&`, `|`, `^`, `<<`, and `>>` with constant folding.
- **Modulo Operator**: Support for `%` operator for integer remainder.
- **Compound Assignment**: Support for `+=`, `-=`, `*=`, `/=`, `%=`, `&=`, `|=`, `^=`, `<<=`, and `>>=`.
- **`for` Loop Declarations**: Support for C99-style variable declarations in the `for` loop initializer.
- **Type Definitions**: Support for `typedef` to create type aliases.
- **Enumerations**: Support for `enum` types and constants.
- **Explicit Casts**: Support for C-style cast expressions `(type)expr` for explicit type conversions, including narrowing, widening, and pointer casts.
- **Implicit Narrowing Warnings**: Compile-time warnings when implicit conversions lose data (e.g., `int` to `char`). Explicit casts suppress the warning.
- **Register Variables**: The `register` storage class allocates local variables in zero page for faster access (shorter instructions, fewer cycles). Falls back to stack if ZP is exhausted.
- **Inline Functions**: The `inline` keyword is accepted (C99). Currently a no-op — functions are called normally via `jsr`.
- **Variadic Functions**: Full support for variadic function declarations (`int printf(const char *fmt, ...)`) and the `<stdarg.h>` interface (`va_list`, `va_start`, `va_arg`, `va_end`). Variadic calls push arguments right-to-left so named parameters have fixed stack offsets. All variadic arguments are promoted to 16-bit per C default argument promotions. `va_arg` reads values via ZP-indirect addressing and advances the `va_list` pointer.
- **Function Pointers**: Declaration, assignment, and indirect call of function pointer types. Syntax: `int (*fp)(int, int)`, `typedef int (*binop_t)(int, int)`, `fp(args)`. Indirect calls use `JSR ($ZP)` (45GS02 absolute-indirect JSR). Function names used as values emit their address into A:X.
- **Constant Propagation**: Substitutes variables with known constant values into expressions at compile time. The optimizer is **branch-aware**, performing constant state merging (intersection) at `if`/`else` boundaries to safely invalidate variables reassigned in branches.
- **ZP-Safe Standard Library**: All standard library functions are designed to be disjoint from the compiler's temporary register pool ($02-$0A). Functions utilize the 45GS02 hardware stack and the Z-scratch restoration pattern (`plz/stz`) to preserve both zero page integrity and 32-bit return values (`AXYZ`).
- **ZP Calling Convention**: Optional zero-page parameter passing (`-fzpcall`) for faster function calls. Parameters are placed in a fixed ZP region instead of the stack, eliminating TSX overhead. Includes caller-save semantics, address-of spill, and per-function metadata (`.zp_uses`, `.zp_clobbers`, `.reg_clobbers`) in `.o45` objects for interprocedural optimization.

## Language Syntax (BNF)

### Statements
The compiler supports a variety of C statements for program flow:

```bnf
<statement> ::= <compound-statement>
              | <expression-statement>
              | <selection-statement>
              | <iteration-statement>
              | <jump-statement>
              | <declaration-statement>
              | <static-assert-statement>
              | <typedef-definition>
              | <enum-definition>

<compound-statement> ::= "{" <block-item-list>? "}"
<block-item-list>    ::= <block-item> <block-item-list>?
<block-item>         ::= <statement> | <declaration-statement> | <typedef-definition> | <enum-definition>

<expression-statement> ::= <expression>? ";"

<selection-statement> ::= "if" "(" <expression> ")" <statement> ( "else" <statement> )?
                        | "switch" "(" <expression> ")" <statement>
                        | "case" <expression> ( "..." <expression> )? ":" <statement>?
                        | "default" ":" <statement>?

<iteration-statement> ::= "while" "(" <expression> ")" <statement>
                        | "do" <statement> "while" "(" <expression> ")" ";"
                        | "for" "(" ( <expression-statement> | <declaration-statement> ) <expression>? ";" <expression>? ")" <statement>
                        | "repeat" "(" <integer-literal> ")" <statement>
                        | "repeat" "(" <type-specifier> <identifier> "," <integer-literal> ")" <statement>

<jump-statement> ::= "return" <expression>? ";"
                   | "break" ";"
                   | "continue" ";"
                   | "continue" <expression> ";"
                   | "continue" "default" ";"
                   | "asm" "(" <string-literal> ")" ";"
                   | "__asm__" "(" <string-literal> ")" ";"

<declaration-statement> ::= ( "_Alignas" "(" ( <expression> | <type-specifier> "*"* ) ")" )? ( "volatile" | "register" )? ( "auto" )? <type-specifier> "*"* <identifier> ( "[" <integer-literal> "]" )? ( "=" <expression> )? ";"
<static-assert-statement> ::= "_Static_assert" "(" <expression> "," <string-literal> ")" ";"
<typedef-definition>    ::= "typedef" <type-specifier> "*"* <identifier> ";"
<enum-definition>       ::= "enum" <identifier>? "{" <enumerator-list> "}" ";"
<enumerator-list>       ::= <enumerator> ( "," <enumerator> )*
<enumerator>            ::= <identifier> ( "=" <integer-literal> )?
<struct-definition>     ::= ( "struct" | "union" ) <identifier>? "{" <member-list> "}" ";"
<member-list>           ::= <member-declaration> <member-list>?
<member-declaration>    ::= ( "_Alignas" "(" ( <expression> | <type-specifier> "*"* ) ")" )? <type-specifier> "*"* <identifier>? ( "[" <integer-literal> "]" )? ";"
                        | <struct-definition>
<type-specifier>        ::= "int" | "char" | "void" | ( "struct" | "union" | "enum" ) <identifier> | <identifier>
```

### Expression Evaluation
Expressions are evaluated using standard C precedence rules:

```bnf
<expression>     ::= <assignment>
<assignment>     ::= <unary> <assignment-op> <assignment> | <conditional>
<assignment-op>  ::= "=" | "+=" | "-=" | "*=" | "/=" | "%=" | "&=" | "|=" | "^=" | "<<=" | ">>="
<conditional>    ::= <logical-or> ( "?" <expression> ":" <conditional> )?
<logical-or>     ::= <logical-and> ( "||" <logical-and> )*
<logical-and>    ::= <bitwise-or> ( "&&" <bitwise-or> )*
<bitwise-or>     ::= <bitwise-xor> ( "|" <bitwise-xor> )*
<bitwise-xor>    ::= <bitwise-and> ( "^" <bitwise-and> )*
<bitwise-and>    ::= <equality> ( "&" <equality> )*
<equality>       ::= <relational> ( ("==" | "!=") <relational> )*
<relational>     ::= <shift> ( ("<" | ">" | "<=" | ">=") <shift> )*
<shift>          ::= <additive> ( ("<<" | ">>") <additive> )*
<additive>       ::= <multiplicative> ( ("+" | "-") <multiplicative> )*
<multiplicative> ::= <unary> ( ("*" | "/") <unary> )*
<unary>          ::= ( "!" | "~" | "-" | "*" | "&" | "++" | "--" ) <unary>
                   | "(" <type-specifier> "*"* ")" "{" <initializer-list> "}"
                   | "(" <type-specifier> "*"* ")" <unary>
                   | <primary> ( "++" | "--" )
                   | <primary> "[" <expression> "]"
                   | <primary>
<primary>        ::= <identifier>
                   | <integer-literal>
                   | <string-literal>
                   | <function-call>
                   | <generic-selection>
                   | "_Alignof" "(" <type-specifier> "*"* ")"
                   | <primary> "." <identifier>
                   | <primary> "->" <identifier>
                   | "(" <expression> ")"

                   <generic-selection> ::= "_Generic" "(" <expression> "," <generic-assoc-list> ")"
                   <generic-assoc-list> ::= <generic-association> ( "," <generic-association> )*
                   <generic-association> ::= ( <type-specifier> "*"* | "default" ) ":" <expression>
                   ```
<function-call> ::= <identifier> "(" <argument-list>? ")"
                   | "__builtin_va_start" "(" <expression> "," <identifier> ")"
                   | "__builtin_va_arg" "(" <expression> "," <type-specifier> "*"* ")"
                   | "__builtin_va_end" "(" <expression> ")"
<argument-list> ::= <expression> ( "," <expression> )*

<function-declaration> ::= <type-specifier> "*"* <identifier> "(" <parameter-list>? ")" ( <compound-statement> | ";" )
<parameter-list>       ::= <parameter> ( "," <parameter> )* ( "," "..." )?
                         | "void"
```

## Statement Descriptions

### Selection Statements (`if`, `else`, `switch`)
Used for conditional branching. 
- **`if` / `else`**: If the condition evaluates to a non-zero value, the first statement is executed. If an `else` clause is present and the condition is zero, the second statement is executed.
- **`switch`**: Evaluates an expression and transfers control to the `case` label with a matching constant value.
    - **`case <const>`**: Defines a jump target for a specific value.
    - **`case <min> ... <max>`**: (Extension) Defines a jump target for a range of values (inclusive). Uses GCC range syntax. Example: `case 'A' ... 'Z':` matches all uppercase letters. Works with `char`, `int`, and `long` types. Multiple ranges and single cases can be mixed freely within the same switch.
    - **`default`**: (Optional) Defines the jump target if no cases match.
    - **Fall-through**: Control flows from one case into the next unless interrupted by a `break` statement.
    - **`continue <value> / default`**: (Extension) Jumps to the specified `case` or `default` label within the *current* `switch` block. This can be used to re-execute logic or share code between cases without using `goto`.

### Iteration Statements (`while`, `do`, `for`, `repeat`)
- **`while`**: Repeatedly executes the body as long as the condition is non-zero. The condition is checked before each iteration.
- **`do`**: Repeatedly executes the body as long as the condition is non-zero. The condition is checked *after* each iteration, ensuring the body executes at least once.
- **`for`**: A flexible loop. The initializer is executed once. The condition is checked before each iteration. The increment expression is executed after each iteration of the body.
- **`repeat`**: (Extension) Compile-time loop unrolling. The body is duplicated N times with zero loop overhead — no branches, no counter, no condition checks. The count must be a compile-time integer constant (0–1024).
    - **Simple form**: `repeat(N) { body }` — unrolls body N times.
    - **Variable form**: `repeat(type var, N) { body }` — unrolls N times with `var` taking values 1, 2, ..., N. The variable is available inside the body as a compile-time constant per iteration.
    ```c
    // Unroll 8 byte copies — no loop overhead
    repeat(8) { *dst++ = *src++; }

    // Initialize with index: buf[0]=1, buf[1]=2, buf[2]=3, buf[3]=4
    repeat(unsigned char x, 4) { buf[x - 1] = x; }
    ```

#### Loop Control: `break` and `continue`
`cc45` supports standard C loop control statements `break` and `continue` within `while`, `do-while`, and `for` loops.

**`break` Statement**
The `break` statement terminates the execution of the innermost enclosing loop. Control passes to the statement immediately following the loop body.

*Example: Searching*
```c
int find_first_even(int* arr, int size) {
    int i;
    int result = -1;
    for (i = 0; i < size; i = i + 1) {
        if (arr[i] % 2 == 0) {
            result = arr[i];
            break; // Exit the for loop immediately
        }
    }
    return result;
}
```

**`continue` Statement**
The `continue` statement skips the remainder of the current iteration of the innermost enclosing loop and proceeds to the next iteration.
- In a **`while`** or **`do-while`** loop, it jumps to the condition evaluation.
- In a **`for`** loop, it jumps to the increment expression.

*Example: Filtering*
```c
int sum_odds(int limit) {
    int i;
    int sum = 0;
    for (i = 0; i < limit; i = i + 1) {
        if (i % 2 == 0) {
            continue; // Skip even numbers
        }
        sum = sum + i;
    }
    return sum;
}
```

**Nested Loops**
`break` and `continue` only affect the innermost loop in which they appear.
```c
for (i = 0; i < 10; i = i + 1) {
    while (j < 10) {
        if (condition) break; // Exits the while loop, but stays in the for loop
        j = j + 1;
    }
}
```

### Jump Statements (`return`, `break`, `continue`, `asm`, `goto`)
- **`return`**: Exits the current function and returns the value of the optional expression to the caller. On Mega65, the return value is passed via the `A` and `X` registers.
- **`break` / `continue`**: Controls the flow of loops (see [Loop Control](#loop-control-break-and-continue) for details).
- **`goto <label>`**: Transfers control unconditionally to the specified label within the current function.
- **`<label>:`**: Defines a jump target for `goto` statements. Labels are function-scoped.
- **`asm` / `__asm__`**: Inserts a string of raw assembly directly into the compiler's output. Each `__asm__("...")` call emits one line of assembly. See [Inline Assembly and Variable Access](#inline-assembly-and-variable-access) for detailed usage.

### Inline Assembly and Variable Access

The `__asm__("...")` statement inserts a single line of assembly into the compiler's output. Each call emits exactly one assembler instruction or directive. Inline assembly can reference C variables using the compiler's naming conventions, giving direct access to parameters, locals, and globals from hand-written assembly.

#### Variable Naming Prefixes

The compiler mangles C variable names to avoid collisions with CPU registers and assembler keywords:

| Scope | Prefix | Storage | Addressing | Example |
|-------|--------|---------|------------|---------|
| Parameter (default) | `_p_` | Hardware stack | Stack-relative (`, s`) | `_p_val` |
| Parameter (`-fzpcall`) | `_p_` | Zero page | Direct ZP (`$XX`) | `_p_val` at `$03` |
| Local | `_l_` | Hardware stack | Stack-relative (`, s`) | `_l_count` |
| Register local | `_l_` | Zero page | Direct ZP (`$XX`) | `_l_i` at `$02` |
| Global | `_` | Data segment | Absolute | `_total` |

#### Accessing Global Variables

Global variables are stored at fixed addresses in the `.data` segment. Access them with absolute addressing — no `, s` suffix needed:

```c
volatile int g_value = 0;

void store_to_global(int val) {
    // Load 16-bit parameter from stack, store to global
    __asm__("ldax _p_val, s");
    __asm__("stax _g_value");
}
```

The compiler prepends `_` to the C variable name: `g_value` becomes `_g_value` in assembly. For a variable named `total`, the assembly name would be `_total`. This follows the traditional C linkage convention (single leading underscore).

8-bit globals use `lda`/`sta` instead of `ldax`/`stax`:

```c
volatile char g_flag = 0;

void set_flag(char f) {
    __asm__("lda.sp _p_f");     // 8-bit stack-relative load
    __asm__("sta _g_flag");     // 8-bit absolute store
}
```

#### Accessing Parameters (Stack-Relative — default)

Function parameters are accessed using the same stack-relative addressing as local variables. The compiler tracks parameter offsets via `.var` directives, automatically adjusting them as local variables are declared. For 16-bit (`int`, pointers) parameters, use `ldax`/`stax`:
```c
void process(int value) {
    __asm__("ldax _p_value, s");   // Load 16-bit param into A (low) / X (high)
    // ... use AX ...
}
```

For 8-bit (`char`) parameters, use `lda.sp`/`sta.sp`:
```c
void process_byte(char b) {
    __asm__("lda.sp _p_b");   // Load 8-bit param into A
    // ... use A ...
}
```

Parameters use the same `TSX; LDA __sp_base+offset,X` sequence as locals. Their offsets are past the return address (2 bytes) and all local variables.

#### Accessing Parameters (ZP — with `-fzpcall`)

When compiled with `-fzpcall`, parameters live at fixed zero-page addresses instead of the stack. The ZP param block starts at `$03` (configurable). Parameters are packed left-to-right:

```c
// -fzpcall: a at $03/$04, b at $05/$06
int add(int a, int b) {
    __asm__("lda $03");    // a (low byte) — direct ZP access
    __asm__("ldx $04");    // a (high byte)
    // No TSX needed — ZP is always accessible
}
```

**Important:** If your function makes nested calls, the compiler automatically saves and restores ZP params around each `jsr`. Inline assembly should not assume ZP param values survive across function calls.

#### Accessing Local Variables (Stack-Relative)

Local variables live on the stack and are accessed via SP-relative addressing (synthesized as `TSX; LDA/STA __sp_base+offset,X`). The compiler tracks their offsets via `.var` directives. Volatile locals are always read from and written to the stack (non-volatile locals may be optimized away):

```c
int copy_value(int val) {
    volatile int result = 0;
    __asm__("ldax _p_val, s");       // Load parameter (stack-relative)
    __asm__("stax _l_result, s");    // Store to local (stack-relative)
    return result;                    // Compiler reads _l_result back
}
```

#### Stack Layout (default convention)

When a function with parameters and locals is executing, the stack looks like this (growing downward):

```
        ┌──────────────────┐  ← Higher addresses
        │   Caller's frame │
        ├──────────────────┤
        │   Parameter N    │  ← _p_paramN (SP-relative)
        │       ...        │
        │   Parameter 1    │  ← _p_param1
        ├──────────────────┤
        │  Return address  │  ← 2 bytes (pushed by JSR)
        ├──────────────────┤
        │   Local var 1    │  ← _l_var1 (SP-relative, offset 0)
        │       ...        │
        │   Local var M    │  ← _l_varM
        └──────────────────┘  ← SP points here
```

The `_fp` assembler variable tracks the current local frame size. It starts at 0 and is automatically adjusted by `.var` directives as locals are pushed. Both parameter and local offsets are SP-relative and shift as new locals are declared — the compiler tracks them all via `.var` bump directives. Inline assembly uses the symbolic names with `, s` and the assembler resolves the correct addressing mode automatically.

#### Stack Layout (with `-fzpcall`)

With ZP calling, parameters are NOT on the stack — they live in fixed ZP addresses. The stack only holds the return address, pre-allocated locals, and the caller-save area:

```
        ┌──────────────────┐  ← Higher addresses
        │   Caller's frame │
        ├──────────────────┤
        │  Return address  │  ← 2 bytes (pushed by JSR)
        ├──────────────────┤
        │   Local var 1    │  ← .local offsets (pre-allocated)
        │       ...        │
        │   Local var M    │
        ├──────────────────┤
        │  Caller-save ZP  │  ← saved ZP param slots (for nested calls)
        └──────────────────┘  ← SP points here

Zero Page:
        $08-$0F               Compiler Scratch / Register Variables
        $10+                  Parameter block (configurable via -Dcc45.zeroPageStart)
```

Parameters have fixed ZP addresses that don't shift when locals are declared. The caller-save area holds copies of the function's own ZP params, saved before nested calls and restored after.

## Memory Layout and Linker Symbols

The CRT startup and linker use several well-known symbols to configure the memory layout of a compiled program. All symbols below can be overridden at link time (via `ln45 -D` or by providing strong definitions in your code). Weak defaults are provided where noted.

### Stack Symbols

| Symbol | Default | Description |
|--------|---------|-------------|
| `__sp_base` | `$0101` | Base address for stack-relative addressing. All stack accesses are synthesized as `TSX; LDA __sp_base+offset,X`. The `+1` accounts for the 6502 SP convention (SP points to the next free byte, not the top of stack). |

By default, the 45GS02 hardware stack lives on page `$01` (`$0100`–`$01FF`), giving 256 bytes of stack space. To relocate the stack to a different (or larger) region:

```c
#pragma crt no_0100_stack
```

This emits `.extern __sp_base` in relocatable mode, allowing the linker to resolve the stack base to any address. Combined with the 45GS02's 16-bit SP (via `CLE` mode), the stack can span a full 64 KB region.

### BSS Symbols

| Symbol | Default | Description |
|--------|---------|-------------|
| `__bss_start` | Auto-generated | Start address of the BSS (uninitialized data) segment. Emitted by the compiler at the beginning of the `.bss` section. |
| `__bss_end` | Auto-generated | End address of the BSS segment. Emitted after the last `.res` directive in `.bss`. |

The CRT startup routine `_init_bss` zeroes all memory from `__bss_start` to `__bss_end` before calling `main`. This ensures uninitialized global variables start at zero, as required by the C standard. To skip BSS zeroing (e.g., for faster startup when globals are explicitly initialized):

```c
#pragma crt no_bssinit
```

### Heap Symbols

| Symbol | Default | Description |
|--------|---------|-------------|
| `__heap_start` | `__bss_end` | Start address of the heap region. Must be 2-byte aligned (rounded up automatically by `_heap_init`). |
| `__heap_end` | `$D000` | End address of the heap region (exclusive). Defaults to the start of the MEGA65 I/O area. |
| `__heap_ptr` | BSS variable | Internal: head of the free list (managed by `malloc`/`free`). |

The heap is only available when enabled via:

```c
#pragma crt heap
```

This links the `crt_heap.o45` CRT module, which calls `_heap_init` during startup. The heap allocator uses a sorted singly-linked free list with block coalescing. Block headers are 4 bytes (2-byte size with allocated bit, 2-byte next pointer). Minimum allocation is 4 bytes, 2-byte aligned.

Both `__heap_start` and `__heap_end` are declared as weak symbols in `malloc.s`, so they can be overridden by strong definitions at link time:

```bash
# Place the heap in a specific region
ln45 -D__heap_start=$8000 -D__heap_end=$C000 main.o45 -l c45.lib -o out.prg
```

The default heap region (`__bss_end` to `$D000`) provides up to ~45 KB for a typical program loaded at `$2000`.

### CRT Startup Sequence

When a `main` function is present, the compiler emits an inline CRT stub with the following call order:

```
__init:
    jsr _init_bss           ; Zero BSS segment (unless no_bssinit)
    jsr _init_heap_crt      ; Initialize heap (only if #pragma crt heap)
    jsr _init_features      ; User-overridable init hook (weak, default RTS)
    jmp _main               ; Enter user program
```

After `_main` returns, control reaches `__exit`, which behaves according to the selected exit mode (`exit_rts`, `exit_halt`, or `exit_brk`).

The `_init_features` symbol is a weak stub (just `RTS`) that can be overridden with a strong definition to add custom initialization (e.g., hardware setup, interrupt vectors).

### Default Memory Map

For a flat binary loaded at `$2000` (the default `.org`):

```
$0000 ┌─────────────────────┐
      │ Zero Page (CPU)     │  ZP $02+ used by compiler (temps + register vars)
$0100 ├─────────────────────┤
      │ Hardware Stack      │  __sp_base = $0101 (default)
$0200 ├─────────────────────┤
      │ System / free       │
$2000 ├─────────────────────┤
      │ CRT startup stub    │  __init, __exit, _init_bss, etc.
      │ Code (.code)        │  Function bodies
      │ Data (.data)        │  Initialized globals, string pool
      │ BSS  (.bss)         │  __bss_start .. __bss_end
      │ Heap                │  __heap_start .. __heap_end
      │   (grows upward)    │
$D000 ├─────────────────────┤  Default __heap_end
      │ I/O registers       │  MEGA65 hardware
$E000 ├─────────────────────┤
      │ KERNAL ROM          │
$FFFF └─────────────────────┘
```

### Relocated Stack Example

With `#pragma crt no_0100_stack`, the stack can be moved out of the 256-byte page-$01 region. This example places a 4 KB stack immediately below the I/O area, with the heap ending where the stack begins:

```c
#pragma crt no_0100_stack
#pragma crt heap
```

```bash
ln45 -D__sp_base=$C001 -D__heap_end=$C000 main.o45 -l c45.lib -o out.prg
```

The 45GS02's 16-bit SP (enabled by `CLE` mode) grows downward from `$CFFF`. `__sp_base` is set to `$C001` (base + 1, matching the 6502 SP convention where SP points to the next free byte).

```
$0000 ┌─────────────────────┐
      │ Zero Page (CPU)     │  ZP $02+ used by compiler (temps + register vars)
$0100 ├─────────────────────┤
      │ (page $01 unused)   │  Free for other uses
$0200 ├─────────────────────┤
      │ System / free       │
$2000 ├─────────────────────┤
      │ CRT startup stub    │  __init, __exit, _init_bss, etc.
      │ Code (.code)        │  Function bodies
      │ Data (.data)        │  Initialized globals, string pool
      │ BSS  (.bss)         │  __bss_start .. __bss_end
      │ Heap                │  __heap_start .. __heap_end
      │   (grows upward)    │
$C000 ├─────────────────────┤  __heap_end
      │ Stack (4 KB)        │  __sp_base = $C001, grows downward
      │   (grows downward)  │  SP starts at $CFFF
$D000 ├─────────────────────┤
      │ I/O registers       │  MEGA65 hardware
$E000 ├─────────────────────┤
      │ KERNAL ROM          │
$FFFF └─────────────────────┘
```

This layout gives the stack 4,096 bytes (vs. 256 on page $01), suitable for deep recursion or programs with many local variables. The heap and stack grow toward each other — the program crashes if they collide (no guard page).

#### Important Caveats

1. **Do not modify SP between reads.** Inline `push`/`pop` instructions change SP, which invalidates all symbolic stack offsets. Load all needed parameters into registers before pushing anything:
   ```c
   // WRONG: push changes SP, making _p_b offset incorrect
   __asm__("ldax _p_a, s");
   __asm__("push .ax");          // SP shifts!
   __asm__("ldax _p_b, s");     // reads wrong location

   // CORRECT: load both params first, then operate
   __asm__("ldax _p_a, s");
   __asm__("stax $08");          // save to ZP temp (no SP change)
   __asm__("ldax _p_b, s");     // SP unchanged, correct offset
   ```

2. **Register clobbering.** After `__asm__`, the compiler assumes all registers are unknown (`invalidateRegs()`). The compiler will reload any values it needs from the stack.

3. **ZP temporaries and register variables.** The compiler uses ZP addresses starting at `$08` (configurable via `-Dcc45.zeroPageStart`) for intermediate results and `register`-qualified local variables. The initial pool is 8 bytes (`$08`–`$0F`) but grows dynamically as needed. Simulated opcodes also use ZP `$08`–`$0F` as scratch space. Avoid relying on ZP values persisting across inline assembly boundaries. If a function uses `register` variables, their ZP addresses may overlap with addresses you expect to be free — check the `; register _l_name = $XX` comments in the generated assembly.

4. **Struct members.** Struct members are not emitted as individual labels. Access a member by adding its byte offset to the base variable name:
   ```c
   // struct { char id; int value; } s; — id at offset 0, value at offset 1
   __asm__("lda.sp _l_s+0");    // s.id (8-bit)
   __asm__("ldax _l_s+1, s");   // s.value (16-bit)
   ```

5. **`@` labels.** Labels within inline assembly can use the `@` prefix (e.g., `@loop:`) to signal internal branch targets. Unlike standard labels, `@` labels do not reset the assembler's register tracking, allowing better optimization.

### Compound Statements (`{ ... }`)
Allows grouping multiple statements into a single block, which is essential for loop bodies and conditional branches.

## Alignment
`cc45` supports the C11 alignment keywords to manage how data is positioned in memory.

### Packed Structs (Default)

All structs are **packed by default** — no alignment padding is inserted between members. This is appropriate for the 45GS02, which has no alignment penalties for unaligned access. The result is minimal memory usage and predictable layouts matching hardware register maps.

```c
struct Packed {
    char a;    // offset 0
    int b;     // offset 1 (no padding after char)
};  // sizeof = 3
```

Use `__unpacked` to opt-in to traditional C alignment padding:

```c
struct __unpacked Padded {
    char a;    // offset 0
    int b;     // offset 2 (1 byte padding after char for int alignment)
};  // sizeof = 4
```

`_Alignas` on individual struct members is always honored, regardless of packed/unpacked mode.

### `_Alignas`
The `_Alignas` specifier can be used to request a specific alignment (in bytes) for global variables and struct members. The alignment must be a power of two.

- **Global Variables**: `_Alignas(16) int buffer[256];` ensures the buffer starts at an address divisible by 16. The compiler emits a `.align` directive to the assembler.
- **Struct Members**: `_Alignas` on a member forces padding to be inserted before that member to satisfy the alignment requirement, even in packed (default) structs.

### `_Alignof`
The `_Alignof` operator returns the alignment requirement of its operand type as a constant of type `int`.

```c
int a = _Alignof(int);   // a = 2
int b = _Alignof(char);  // b = 1
```

## Type System
- **`int`** / **`short`**: 16-bit unsigned word. `short` is an alias for `int` on this 16-bit target. Arithmetic and comparisons are performed using 16-bit logic.
- **`char`**: 8-bit unsigned byte. Promoted to 16-bit during most expression evaluations to maintain consistency.
- **`_Bool`**: 8-bit boolean type (C99). Stores only `0` or `1` — any non-zero value assigned or cast to `_Bool` is normalized to `1` at both compile time (constant folding) and runtime. `sizeof(_Bool)` is 1. The `<stdbool.h>` header provides `bool`, `true`, and `false` macros.
- **`struct` & `union`**: Custom aggregate types.
    - **`struct`**: Members are allocated sequentially in memory with appropriate padding for alignment.
    - **`union`**: All members share the same base address, allowing different interpretations of the same memory. The total size is the size of the largest member, padded for max alignment.
- **Anonymous Aggregates**: Nested `struct` or `union` definitions without a member name. Their members are "flattened" into the parent scope, allowing direct access (e.g., `s.inner_member`).
- **`void`**: Incomplete type used only as a function return type (`void func()`) or as a generic pointer (`void *`). `void *` is a 16-bit pointer that can point to any type, and is accepted in function parameters, return types, local variables, and cast expressions. Arithmetic on `void *` is not supported — cast to a concrete pointer type first.
- **Pointers**: All pointers are 16-bit addresses. The compiler supports multiple levels of indirection (e.g., `char **pp`).
    - **Dereferencing**: Standard `*` operator for accessing memory.
    - **Address-of**: Standard `&` operator for obtaining the stack-relative address of a variable.
    - **Arithmetic**: Supports pointer addition and subtraction, scaled by the size of the pointed-to type.
- **`volatile`**: Prevents the compiler from optimizing away reads or writes to the variable. Essential for I/O registers or variables shared with interrupt handlers.
- **`auto`**: (Classic C) Declares a variable with automatic storage duration. In `ccomp`, all local variables are automatic by default, so this keyword is functionally a no-op provided for compatibility.
- **`register`**: Requests zero-page allocation for a local variable. ZP access is faster than stack-relative (shorter instructions, fewer cycles). Falls back to stack allocation if ZP space is exhausted. See [Storage Classes](#register) for details.
- **`unsigned`**: Explicitly declares an unsigned integer type. Supported as `unsigned int` (16-bit), `unsigned char` (8-bit), or a bare `unsigned` (equivalent to `unsigned int`).
- **`signed`**: Explicitly declares a signed integer type. Supported as `signed int` (16-bit) and `signed char` (8-bit). Signed types use two's complement representation. Relational operators (`<`, `>`, `<=`, `>=`) are signedness-aware when operating on these types.

### `sizeof` Operator
The `sizeof` operator returns the size, in bytes, of its operand. The operand can be a type name (e.g., `sizeof(int)`) or an expression (e.g., `sizeof x`). The result is a constant of type `int`.

```c
int a = sizeof(int);      // a = 2
int b = sizeof(char);     // b = 1
int b2 = sizeof(_Bool);  // b2 = 1
struct Point { int x, y; };
int c = sizeof(struct Point); // c = 4
```

> **Note on Padding**: `cc45` may insert padding in structs to satisfy alignment requirements, which will be reflected in the `sizeof` result.

### Cast Expressions
The cast operator `(type)expr` performs an explicit type conversion. The compiler supports casts between all scalar types:

```c
int big = 0x1234;
char low = (char)big;      // Narrowing: keeps low byte (0x34)
char c = 200;
int wide = (int)c;         // Widening: zero-extends to 16-bit (200)
char *p = &c;
int addr = (int)p;         // Pointer to int
_Bool b = (_Bool)big;      // Bool normalization: non-zero -> 1
```

**Narrowing casts** (int/pointer to char) discard the high byte, keeping only the low 8 bits. **Widening casts** (char to int/pointer) zero-extend the value to 16 bits.

Cast expressions are folded at compile time when the operand is a constant (e.g., `(char)0x1FF` becomes `0xFF`).

### Implicit Narrowing Warnings
When an implicit conversion loses data, the compiler emits a warning to stderr:
```
file.c:8:5: warning: implicit conversion loses data from 'int' to 'char'
```
This occurs when assigning an `int` or pointer value to a `char` variable without an explicit cast. Using a cast expression suppresses the warning, signalling that the truncation is intentional.

### Unused Variable Warnings
The compiler issues warnings for local variables and function parameters that are declared but never referenced in any expression:
```
file.c:4:9: warning: unused variable 'y'
```
This check is integrated with the `ConstantFolder` to ensure that variables used in expressions that are subsequently constant-folded are not incorrectly flagged as unused.

> **Note on Declaration Order**: Type qualifiers (`volatile`, `const`, `signed`, `unsigned`) can appear in any order in declarations. For example, `volatile signed int x`, `int volatile x`, `signed volatile int x`, and `unsigned const char c` are all valid and equivalent.

## Optimizations
- **Register Tracking**: The compiler tracks the state of `A`, `X`, `Y`, and `Z` registers to eliminate redundant loads and stores.
    - **Granular 16-bit Operations**: For 16-bit (`int`, pointer) variables, the compiler tracks the low and high bytes independently in the `A` and `X` registers. It can skip loading or storing individual bytes if they are already in the correct state.
    - **Self-Assignment Elimination**: Standalone statements like `x = x;` are completely omitted for non-volatile variables.
- **Zero Page Cache**: Intermediate expression results are cached in a pool of Zero Page registers, reducing stack traffic. The `register` keyword allocates local variables in ZP for even faster access.
- **Register Variables**: The `register` storage class allocates local variables in zero page instead of on the stack. ZP instructions are shorter (2 bytes vs 3) and faster. See [Storage Classes](#register).
- **Direct Stack Access**: Access to local variables, parameters, and struct members on the stack is synthesized using `TSX` + absolute,X indexed addressing (e.g., `TSX; LDA __sp_base+3,X`), since the 45GS02 has no native direct stack-relative load/store instructions. The `__sp_base` symbol defaults to `$0101` (page $01, +1 for SP convention) and can be relocated via `#pragma crt no_0100_stack` for non-standard stack pages.
- **Push/Pop Abstractions**: Utilizes the assembler's high-level `push` and `pop` instructions for 8-bit and 16-bit register saves/restores, improving code readability.
- **Strength Reduction**: Converts expensive operations into faster equivalents (e.g., `x * 2` becomes `x << 1`, `x % 8` becomes `x & 7`).
- **Constant Propagation**: Substitutes variables with known constant values into expressions at compile time. The optimizer is **branch-aware**, performing constant state merging (intersection) at `if`/`else` boundaries to safely invalidate variables that are reassigned in only one execution path.
- **Constant Folding**: Arithmetic and logical expressions with literal or propagated constant operands are evaluated at compile time. The folder preserves the original C type (e.g., `long`) for folded results to ensure correct stack-push sizing for variadic functions.
- **Dead Variable Elimination**: Removes stack allocation and initialization for local variables that are initialized with constants and not subsequently used (skips `volatile` variables).
- **Dead Code Removal**: Detects and removes code following a `return` statement.
- **Segment-based Separation**: Automatically organizes output into `code`, `data`, and `bss` segments. Initialized globals are placed in `data`, uninitialized globals in `bss`, and function bodies in `code`, allowing for optimized memory layout by the assembler.
- **Logical Short-circuiting**: Optimizes logical `&&` and `||` operators to jump directly to target labels, bypassing intermediate boolean conversions.
- **Increment Optimization**: Uses `INC A`, `INX`, `INW`, etc., for `+ 1` operations.
- **Tiered Branching**: Automatically selects between short (8-bit) and long (16-bit) relative branches.
- **Tail-Call Optimization**: When a function's last action is calling another function (`JSR label; RTS`), the assembler optimizer converts this to a single `JMP label`, saving 1 byte and 12 cycles (JSR+RTS overhead). Only applies to plain `RTS` (not `RTS #n` with stack cleanup). Critical for recursive algorithms on the 256-byte hardware stack.
- **Fused Compare-and-Branch**: When a comparison result (`<`, `>`, `==`, `!=`, etc.) is immediately tested by a conditional branch, the compiler emits a direct branch instruction (`bcc`, `bcs`, `beq`, `bne`) on the CPU flags instead of materializing a boolean value (0 or 1) into a register and testing it separately.
- **AND/OR Branch Fusion**: When an AND, OR, or XOR result is tested for truthiness in an `if` condition, the Z flag set by the bitwise instruction is used directly for branching (`bne`/`beq`), skipping the boolean materialization.
- **BBS/BBR Single-Bit Tests**: When `if (var & power_of_2)` tests a single bit and the variable is in zero page, the compiler emits a `BBS`/`BBR` instruction (3 bytes, no register clobber) instead of loading, masking, and branching.
- **Branch Inversion**: The assembler optimizer detects `Bcc +2; BRA target` patterns (conditional branch skipping over an unconditional branch) and inverts the condition to branch directly to the target, eliminating the `BRA` instruction (saves 2 bytes per occurrence).
- **Pointer Constant Propagation**: When a pointer variable holds a known constant address (e.g., `unsigned char *p = (unsigned char *)0xD020`), stores through the pointer (`*p = val`) emit direct absolute addressing (`sta $D020`) instead of loading the pointer and using indirect addressing.
- **Dead Local Variable Elimination**: After all IR instructions are generated for a function, local variables that are written but never read are detected and their initialization code is removed. Handles transitive chains (e.g., a CONST feeding a dead STORE).
- **8-bit Char Arithmetic**: Operations on `char` variables with small constants stay at 8-bit width (`and`, `ora`, `eor`, `adc`, `sbc`, `cmp`) instead of widening to 16-bit. Char increment/decrement uses the single-byte `INC A`/`DEC A` instructions.
- **Bidirectional Store/Load Tracking**: The assembler optimizer tracks values stored to memory. After `STA $addr`, a subsequent `LDA $addr` is eliminated if A still holds the same value. Also handles cases where A holds an unknown value after an ALU operation — after `STA`, the optimizer records that A and the memory location are equivalent.
- **Dead Trailing Block Elimination**: Unreachable code blocks after `while(1)` infinite loops (such as `return 0` after the loop body) are detected and removed from the IR.

## Intrinsics

cc45 provides high-level access to the MEGA65 processor state through built-in structures.

### CPU Registers (`__cpu`)

You can read and write physical and virtual registers directly. 8-bit registers are type `unsigned char`, 16-bit are `unsigned int`, and 32-bit are `unsigned long`.

| Expression | Type | Description |
| :--- | :--- | :--- |
| `__cpu.A`, `.X`, `.Y`, `.Z` | `unsigned char` | 8-bit physical registers |
| `__cpu.AX`, `.AY`, `.AZ`, `.XY` | `unsigned int` | 16-bit virtual pairs |
| `__cpu.SP` | `unsigned int` | 16-bit Stack Pointer |
| `__cpu.Q`, `.AXYZ` | `unsigned long` | 32-bit virtual quad |

**Example:**
```c
__cpu.A = 0x42;
__cpu.X = __cpu.Y;
__cpu.AX = 0x1234;
```

### Processor Flags (`__flags`)

Processor flags are accessible as `_Bool` values.

| Expression | Type | Description |
| :--- | :--- | :--- |
| `__flags.Carry` | `_Bool` | Carry flag |
| `__flags.Zero` | `_Bool` | Zero flag |
| `__flags.Negative` | `_Bool` | Negative flag |
| `__flags.Overflow` | `_Bool` | Overflow flag |

**Example:**
```c
__flags.Carry = 1;  // SEC
if (__flags.Zero) { // Capture and test zero flag
    ...
}
```

### Safety Restrictions

To prevent incorrect code generation and maintain compiler invariants:
- **No Address-of**: Taking the address of a register or flag (e.g., `&__cpu.A`) is a compile-time error.
- **Volatile Semantics**: The compiler treats these as inherently volatile; they are not subject to standard dead-store elimination.

## Compilation Process
1.  **Preprocessing**: Invokes `cp45` to handle file inclusion, macro definition, and conditional compilation.
2.  **Lexical Analysis**: Strips comments and tokenizes the preprocessed source.
3.  **Parsing**: Builds an AST using recursive descent.
4.  **Constant Folding**: Simplifies the AST by evaluating constant expressions.
5.  **Code Generation**: Traverses the AST and emits high-level `ca45` assembly.
    - Function arguments and local variables are managed via the stack using stack-relative addressing (`offset, s`).
    - The compiler uses the `.var` and `.cleanup` assembler directives to maintain correct stack offsets.

### Implementation Details

### Variable Naming Conventions
To prevent naming collisions between C variables and 45GS02 processor registers (A, X, Y, Z), status flags (P.C, P.Z, etc.), or internal assembler labels, `cc45` uses a prefixing scheme in the generated assembly:
- **Parameters**: Prefixed with `_p_` (e.g., `val` becomes `_p_val`). Declared on the `proc` instruction line with size qualifiers (`W#` for 16-bit, `B#` for 8-bit).
- **Local Variables**: Prefixed with `_l_` (e.g., `i` becomes `_l_i`). Offsets are tracked via `.var` directives as variables are pushed onto the stack.
- **Global Variables**: Prefixed with `_` (e.g., `total` becomes `_total`). Stored in the `.data` segment at absolute addresses. This follows the traditional C linkage convention (single leading underscore), ensuring compatibility with other 6502 toolchains (cc65, etc.).
- **Functions**: Prefixed with `_` (e.g., `main` becomes `_main`). Emitted as `proc _main` and called via `jsr _main`.

These prefixes are applied to both variable declarations and all subsequent instructions referencing those variables. The same names are accessible from `__asm__()` inline assembly (see [Inline Assembly and Variable Access](#inline-assembly-and-variable-access)).

### Storage Classes

#### `static`

The `static` keyword provides two distinct behaviors depending on context:

- **File-scope (globals and functions)**: Restricts the symbol to internal linkage. The variable or function is defined and usable within the translation unit but is NOT exported via `.global` in relocatable mode. This prevents name collisions when linking multiple object files.
  ```c
  static int file_counter = 0;  // not visible to other .o45 files
  static int helper(int x) { return x + 1; }  // not exported
  ```

- **Local variables**: Gives the variable static storage duration. Instead of being allocated on the stack (which is destroyed on function return), the variable is placed in the `.data` or `.bss` segment with a mangled name (`_funcname__varname`). The initial value is set once at program load, and the variable retains its value across function calls.
  ```c
  int counter() {
      static int calls = 0;  // persists across calls
      calls = calls + 1;
      return calls;
  }
  ```

The constant folder treats `static` locals like `volatile` variables — it does not propagate their initial value, since the value may change between calls.

#### `register`

The `register` keyword requests that a local variable be allocated in zero page instead of on the hardware stack. Zero-page access on the 45GS02 is faster than stack-relative access: instructions are 1 byte shorter (2 bytes vs 3) and execute in fewer cycles.

```c
int sum_loop(int n) {
    register int i;
    register int sum = 0;
    for (i = 0; i < n; i = i + 1) {
        sum = sum + i;
    }
    return sum;
}
```

In the generated assembly, `i` and `sum` are allocated at ZP addresses (e.g., `$02` and `$04`) and accessed via direct ZP addressing (`lda $02`, `sta $04`) rather than stack-relative pseudo-ops (`lda.sp _l_i`). Increment optimizations like `inw $02` work naturally with ZP addresses.

**Behavior:**
- The compiler allocates ZP bytes from the same pool used for expression temporaries (`allocateZP`), dynamically growing as needed up to the ZP limit (`$FF`).
- Register variables don't consume stack space — they are not pushed, don't participate in `.var` offset bumping, and don't require cleanup on function return.
- If no ZP space is available, the compiler silently falls back to normal stack allocation. Per the C standard, `register` is a hint — the compiler is free to ignore it.
- Supported in regular declarations and `for` loop initializers (`for (register int i = 0; ...)`).
- Arrays and struct types are excluded from register allocation.
- ZP slots are released at function exit, so register variables in different functions reuse the same ZP addresses.

**Limitations:**
- The address-of operator (`&`) applied to a `register` variable is a compile error (`"Cannot take address of register variable"`), as required by the C standard (§6.7.1).
- Register variables share the ZP pool with expression temporaries. Heavy use of `register` may push temporaries to higher ZP addresses.

#### `inline`

The `inline` keyword is accepted as a function specifier (C99). The compiler parses `inline`, `__inline`, and `__inline__` but currently treats them as no-ops — the function is emitted normally and called via `jsr`. Actual function body inlining is planned for a future release.

```c
inline int square(int x) { return x * x; }
int main() { return square(5); }  // emits jsr _square (not yet inlined)
```

#### `extern`

The `extern` keyword declares a variable or function with external linkage without defining it. The type is recorded but no storage is emitted. In relocatable mode, the symbol is emitted via `.extern`.

#### `__interrupt`

The `__interrupt` function specifier marks a function as an interrupt service routine. The compiler automatically emits register save/restore sequences and uses `RTI` instead of `RTS`:

```c
__interrupt void timer_isr(void) {
    tick_count++;
}
// Emits: PHA/PHX/PHY/PHZ at entry, PLZ/PLY/PLX/PLA/RTI at exit
```

Interrupt functions save and restore all four CPU registers (A, X, Y, Z) unconditionally. This ensures the interrupted code's register state is fully preserved.

#### `__naked`

The `__naked` function specifier suppresses all compiler-generated prologue and epilogue code. The function body is emitted directly between `proc` and `endproc` with no frame setup, no parameter handling, and no return instruction:

```c
__naked void fast_routine(void) {
    __asm__("lda #$42");
    __asm__("sta $0400");
    __asm__("rts");
}
```

Use `__naked` for hand-optimized assembly routines where the compiler's overhead is unacceptable. The programmer is responsible for all register management and returning.

### Label Generation and Scoping
To ensure labels are unique across different functions and scopes, `cc45` utilizes a hierarchical scoping system in the generated assembly:
- **Temporary Labels**: Internal loop and branch targets are generated as `L<n>`.
- **Fully Qualified Names**: When processed by the internal assembler or viewed in an expanded listing (Level 2), these labels are prefixed with their containing procedure or scope name, separated by colons (e.g., `main:L0:`).
- **Scoping**: This prevents collisions between common label names used in different C functions by flattening the hierarchy into globally unique identifiers.

### Major Structures
- **`ASTNode` / `ASTVisitor`**: The core of the compiler's architecture. Every language construct is an `ASTNode`, and `CodeGenerator` is an `ASTVisitor` that traverses the tree to emit code.
- **`RegState`**: A structure used to track the contents of the CPU registers (`A`, `X`, `Y`, `Z`). It stores whether a register's value is currently `known`, if it holds a specific `variable` and `offset`, or a literal `value`.
- **`ZPReg`**: Manages the allocation of virtual zero-page registers used for intermediate expression results.

### Register Tracking
The compiler tracks the state of all four primary 45GS02 registers to eliminate redundant `LDA` and `LDX` instructions.
- **Scope**: Tracking is local to basic blocks. It is automatically invalidated (`invalidateRegs()`) at control flow boundaries such as labels, branches, and function calls.
- **Optimization**: If a variable or constant is already present in a register, the compiler will skip the corresponding load instruction.

### Processor Flag Tracking
The compiler tracks the state of the processor status register (Carry, Zero, Negative, and Overflow flags) to eliminate redundant instructions.
- **TriState Logic**: Flags are tracked using a TriState (`SET`, `CLEAR`, `UNKNOWN`).
- **ZN Source Tracking**: The compiler tracks which register (`A`, `X`, `Y`, or `Z`) was the last to affect the Zero (Z) and Negative (N) flags.
- **Optimizations**:
    - **Carry Flag**: Redundant `CLC` and `SEC` instructions are omitted if the carry flag is already known to be in the required state (e.g., before an `ADC` or `SBC`).
    - **CMP Elimination**: `CMP #$00` instructions are skipped in control flow statements (`if`, `while`, etc.) if the flags already reflect the state of the register holding the condition.
    - **8-bit Control Flow**: Conditions using `char` (8-bit) types are optimized to use a single `BEQ/BNE` instruction, bypassing the 16-bit truthiness check used for `int` and pointers.
- **Invalidation**: Flags are invalidated at control flow boundaries (labels, branches) and after instructions that affect flags in complex ways (like `ADC`, `SBC`, or function calls).


### Adding Custom Opcodes
To add support for a new opcode or addressing mode in the compiler's output:
1.  **`M65Emitter`**: Add a new method to `include/M65Emitter.hpp` and implement it in `src/main/M65Emitter.cpp`.
2.  **`CodeGenerator`**: Call the new emitter method from the appropriate `visit()` function in `src/main/CodeGenerator.cpp`.

## Calling Conventions

`cc45` supports two calling conventions for passing function parameters. Both can coexist in the same program — the convention is selected per-compilation-unit via `-fzpcall` or per-function via `__fastcall__`.

```
                    Stack Convention              ZP Convention
                    (default, -fno-zpcall)        (-fzpcall / __fastcall__)
                    ─────────────────────         ─────────────────────
  Parameters:       Pushed onto hardware stack    Stored in fixed ZP region
  Param access:     TSX + absolute,X indexed      Direct ZP load/store
  Instruction size: 3-4 bytes per access          2 bytes per access
  Cycles per access:5+ cycles                     3 cycles
  Max params:       Limited by stack depth         8 bytes (default, configurable)
  Reentrancy:       Fully reentrant               Not reentrant by default
  Variadic:         Supported                     Falls back to stack convention
  Indirect calls:   Supported                     Falls back to stack convention
  Recursion:        Native (params on stack)       Supported (caller-save to frame)
```

### Stack Convention (Default)

The stack convention is the default. Parameters are pushed onto the 45GS02 hardware stack before `JSR`, and the callee accesses them via SP-relative addressing (`TSX; LDA __sp_base+offset,X`). Arguments are evaluated and pushed left-to-right; each is 2 bytes (`int`, pointer) or 1 byte (`char`, promoted to 2). The caller is responsible for removing arguments from the stack after the call returns (cdecl-style, `.cleanup N`).

**Advantages:** Fully reentrant, no parameter limit, variadic support, function pointer support, simple recursion.

**Limitations:** Every parameter read requires a `TSX` (clobbers X) followed by absolute,X indexed load — 5+ cycles, 3-4 bytes per access.

### ZP Convention (`-fzpcall` / `__fastcall__`)

When enabled, function parameters are passed via a fixed zero-page region instead of the hardware stack. The caller stores arguments into ZP slots before `JSR`; the callee reads them with fast ZP addressing.

#### Enabling ZP Calling

There are two ways to enable ZP calling:

- **Per-compilation-unit**: `-fzpcall` flag applies ZP calling to all functions in the file. All non-variadic, non-indirect calls use ZP params.
- **Per-function**: The `__fastcall__` function specifier marks individual functions for ZP calling, even without `-fzpcall`. This allows mixing conventions within a single file.

```c
// Per-function: only add() uses ZP calling
__fastcall__ int add(int a, int b) {
    return a + b;
}

// mul() uses the default stack calling convention
int mul(int a, int b) {
    return a * b;
}
```

Both the function definition and any prototype declarations should include `__fastcall__` so the caller knows to use ZP param passing.

#### Parameter Layout

Parameters are packed left-to-right into ZP slots starting at `zeroPageStart + 1` (default `$03`):

```
  ZP Address    Contents
  ──────────    ────────────────────────
  $02           __zp_scratch (reserved for simulated ops)
  $03           Parameter byte 0 (first param, lo)
  $04           Parameter byte 1 (first param, hi — if int/ptr)
  $05           Parameter byte 2 (second param, lo)
  $06           Parameter byte 3 (second param, hi — if int/ptr)
  ...           ...
  $0A           Parameter byte 7 (last available with default config)
```

Type packing: `char` → 1 byte, `int`/pointer → 2 bytes (lo, hi), `long` → 4 bytes (little-endian).

#### Calling Sequence

```
  Caller:                          Callee (add):
  ┌───────────────────────┐        ┌───────────────────────┐
  │ lda #<arg1            │        │ ; a at $03/$04        │
  │ sta $03               │        │ ; b at $05/$06        │
  │ lda #>arg1            │        │ lda $03               │
  │ sta $04               │        │ clc                   │
  │ lda #<arg2            │        │ adc $05               │
  │ sta $05               │        │ tax                   │
  │ lda #>arg2            │        │ lda $04               │
  │ sta $06               │        │ adc $06               │
  │ jsr _add              │        │ ; result in AX        │
  │ ; result in AX        │        │ rts                   │
  └───────────────────────┘        └───────────────────────┘
```

#### Caller-Save Semantics

When a function makes nested calls, its own ZP parameters are overwritten by the callee. The compiler automatically generates save/restore sequences around each call site. **Leaf functions** (functions that make no calls) skip save/restore entirely — the compiler detects these automatically.

#### Address-of Parameters

If `&param` is used, the parameter is spilled from ZP to a pre-allocated frame slot at function entry. All subsequent access goes through the frame slot, which has a stable runtime address.

#### Return Values

| Type | Convention |
|------|-----------|
| `char` | A register |
| `int`/pointer | AX registers |
| `long` | AXYZ registers (no hidden pointer needed) |
| `struct` | Hidden pointer in ZP $03/$04; callee writes through it |

Note: `long` returns are more efficient under ZP convention — returned directly in AXYZ, whereas the stack convention uses a hidden-pointer ABI (same as struct returns).

#### Limitations

- Maximum parameter bytes = `zeroPageAvail - 1` (default: 8 bytes). Configurable up to 32 via `-Dcc45.zeroPageAvail=<N>`.
- Variadic functions (`printf`, etc.) automatically fall back to stack convention.
- Indirect calls (function pointers) fall back to stack convention (callee identity unknown at compile time).
- Not reentrant by default — interrupt handlers must save the ZP param region (`$03`–`$0A`).

#### Function Metadata

Each function emits metadata directives describing its resource usage. In `-c` (relocatable) mode, these are stored in the `.o45` symbol table:
- `.zp_uses` — ZP slots read as input parameters (zpcall/fastcall only)
- `.zp_clobbers` — ZP slots this function may overwrite (zpcall/fastcall only)
- `.reg_clobbers` — CPU registers clobbered (subset of A, X, Y, Z)
- `.flag_clobbers` — Processor flags clobbered (subset of C, N, Z, V)

The compiler tracks which registers and flags each function actually modifies during code generation, emitting only the registers/flags that are touched. For example, a leaf fastcall function `int identity(int x) { return x; }` emits `.reg_clobbers A, X` rather than all four registers. This metadata enables interprocedural optimization by the linker and future selective register invalidation at call sites.

### Mixing Conventions

The two conventions coexist in the same program. The compiler automatically handles convention boundaries:

- **ZP function calling variadic function**: the compiler emits stack-push argument passing for the variadic call and saves/restores the caller's ZP params around it.
- **ZP function calling via function pointer**: indirect calls use stack-push convention; the compiler saves/restores ZP params around the call.
- **Stack function calling ZP function**: works when the callee is declared `__fastcall__` (the caller knows to set up ZP params). With `-fzpcall`, the caller must also be compiled in ZP mode.

Each convention has its own standard library and CRT:

| Convention | Library |
|------------|---------|
| Stack (default) | `c45.lib` |
| ZP (`-fzpcall`) | `c45_zp.lib` |

### Variadic Functions

Variadic functions accept a variable number of arguments after some fixed parameters. The `printf` family and user-defined variadic functions use the standard `<stdarg.h>` interface: `va_list`, `va_start`, `va_arg`, and `va_end`.

#### How Variadic Arguments Work

When a function is declared with `...`, the compiler uses **stack convention** for argument passing, even if compiled with `-fzpcall`. This is necessary because the callee cannot know in advance how many arguments the caller will pass.

```c
// Declaration
int printf(const char *fmt, ...);

// Call with 2 variadic args
printf("x=%d y=%d\n", x, y);  
```

Arguments are pushed onto the stack right-to-left (rightmost first), so the named parameters (like `fmt`) have fixed offsets from the stack pointer. The variadic arguments follow:

```
Stack layout after printf call:
        ┌──────────────────┐  ← Higher addresses
        │     Return addr  │  ← Caller's frame
        ├──────────────────┤
        │   arg 2 (y)      │  ← second variadic arg (rightmost)
        │   arg 1 (x)      │  ← first variadic arg
        │   fmt pointer    │  ← fixed named parameter
        ├──────────────────┤
        │   Return address │  ← 2 bytes (pushed by JSR)
        ├──────────────────┤
        │   Local var 1    │  ← Printf's locals
        │       ...        │
        └──────────────────┘  ← SP (inside printf)
```

#### Using `va_list` and `va_arg`

The `<stdarg.h>` interface provides portable access to variadic arguments:

```c
#include <stdarg.h>
#include <stdio.h>

int my_vprintf(const char *fmt, va_list args) {
    int count = 0;
    while (*fmt) {
        if (*fmt == '%' && *(fmt+1) == 'd') {
            int val = va_arg(args, int);  // Next int argument
            printf("%d", val);
            count = count + 1;
            fmt = fmt + 2;
        } else {
            putchar(*fmt);
            fmt = fmt + 1;
        }
    }
    return count;
}

int my_printf(const char *fmt, ...) {
    va_list args;
    va_start(args, fmt);           // Initialize args to point past fmt
    int result = my_vprintf(fmt, args);
    va_end(args);
    return result;
}
```

**Implementation details:**
- `va_list` is a typedef to `unsigned int` (16-bit) holding a stack offset
- `va_start(args, last_named)` initializes `args` to point to the first variadic argument (past `last_named`)
- `va_arg(args, type)` reads the next argument of the given type and advances `args` by that type's size
- `va_end(args)` is a no-op macro for cleanup compatibility

#### Variadic Functions and ZP Calling

When a ZP-convention function (`-fzpcall` or `__fastcall__`) calls a variadic function, the compiler automatically switches to stack argument passing for that call:

```c
__fastcall__ void process(int data) {
    // data is in ZP ($03/$04)
    printf("data=%d\n", data);  // Switches to stack convention for this call
    // After printf returns, ZP params are still intact
}
```

The compiler generates:
1. Stack-push argument passing for `printf`'s arguments
2. Save/restore of the function's own ZP parameters around the call
3. Return to ZP-convention parameter reading after the call

This seamless transition is transparent to the programmer — no explicit code changes needed.

#### Implementation in Assembly

If writing variadic functions in assembly, the parameter layout matches the C calling convention:

```asm
; int my_sum(int count, ...)  — return sum of count following ints
proc my_sum
    ; count is at __sp_base+4, S
    ldax _p_count, s        ; Load count into AX
    
    tsx                     ; Set up for variadic access
    lda __sp_base+2, x      ; First variadic arg (offset +2 from count param)
    clc
    adc __sp_base+4, x      ; Add second variadic arg
    adc __sp_base+6, x      ; Add third variadic arg
    ; result in A
    rts
endproc
```

### Return Value Handling Across Calling Conventions

Return values use the same register convention regardless of calling convention. This ensures safe interoperability between code compiled with different conventions.

#### Return Value Registers (Both Conventions)

| Type | Registers | Notes |
|------|-----------|-------|
| `char` | A | 8-bit value in A, X undefined |
| `int` / pointer | AX | 16-bit value: A = low, X = high |
| `long` | AXYZ | 32-bit value: A = bits 0-7, X = bits 8-15, Y = bits 16-23, Z = bits 24-31 |
| `void` | (none) | No value returned |
| `struct`/`union` | Hidden pointer | Caller passes address in $03/$04 (ZP) or as first stack param (stack); callee writes result there |

#### Safe Mixed-Convention Calls

Return values are safe when mixing calling conventions because both use the same register destinations:

```c
__fastcall__ int add_zp(int a, int b) {  // ZP convention
    return a + b;  // Returns in AX
}

int caller() {           // Stack convention (default)
    int result = add_zp(10, 20);  // Calls ZP function
    // Result safely in AX
    return result;
}
```

The compiler handles this automatically:
1. Caller (stack convention) stores arguments to ZP ($03+ params)
2. Callee (ZP convention) reads from ZP and returns in AX
3. Caller receives result in AX — no special handling needed

#### Enforcement: Why ZP→Stack Calls Are Forbidden

The linker enforces that ZP-convention callers do **not** call stack-convention callees. This is a one-directional restriction:

```c
__fastcall__ int zp_func() { ... }    // ZP convention
int stack_func() { ... }               // Stack convention (default)

__fastcall__ void zp_caller() {
    zp_func();        // OK — both ZP
    stack_func();     // ERROR: ZP caller cannot call stack callee
}
```

**Why?** ZP-convention callers have no mechanism to pass stack parameters:
- Stack callers: push args to hardware stack → return address at top
- Stack callees: read args from stack via TSX offset
- ZP callers: put args in ZP slots → expect callee to read from ZP
- **Result of mismatch**: Callee reads garbage from uninitialized stack offsets

Return values are not the problem — both conventions return in the same registers (AX for `int`, AXYZ for `long`). The problem is **parameter passing**: the ZP caller cannot push arguments to the stack because it doesn't know the calling convention of the callee.

**Stack→ZP calls are allowed** because stack callers can set up ZP slots as easily as any other caller. No extra mechanism needed.

#### Struct Returns: Hidden Pointer Semantics

Structs are returned via a hidden pointer, but the register location differs slightly:

```c
struct Point { int x, y; };

// Stack convention
struct Point get_point_stack() {
    struct Point p = {1, 2};
    return p;  // Hidden pointer in caller's first parameter slot
}

// ZP convention
__fastcall__ struct Point get_point_zp() {
    struct Point p = {1, 2};
    return p;  // Hidden pointer in ZP $03/$04
}

void caller() {
    // Both work; compiler inserts hidden pointer transparently
    struct Point p1 = get_point_stack();
    struct Point p2 = get_point_zp();
}
```

In both cases:
1. Caller allocates space for the struct
2. Caller passes the address (either on stack or in ZP)
3. Callee writes the struct to that address
4. Caller reads the result from the pre-allocated space

Mixing conventions with struct returns is safe because the linker prevents the problematic ZP→stack direction entirely.

### When to Use Each Convention

| Scenario | Recommended Convention |
|----------|----------------------|
| General-purpose code | Stack (default) — safest, most compatible |
| Performance-critical inner loops | ZP (`__fastcall__`) — fewer cycles per param access |
| Interrupt handlers | Stack — reentrant without saving ZP |
| Variadic functions (`printf`, etc.) | Stack — required by the calling mechanism |
| Hand-written assembly libraries | ZP — natural fit for 6502-style ZP usage |
| Programs with many small functions | ZP (`-fzpcall`) — reduced code size from shorter instructions |
| Deep recursion | Stack — no per-call ZP save/restore overhead |
| Whole-program builds | ZP (`-fzpcall`) — linker can optimize save/restore sets |

### Configuring the ZP Region

```bash
# Move ZP region to start at $10 (scratch at $10, params at $11+)
cc45 -fzpcall -Dcc45.zeroPageStart=$10 ...

# Expand to 16 param bytes ($03-$12)
cc45 -fzpcall -Dcc45.zeroPageAvail=17 ...
```

### Convention Comparison

```
  STACK CONVENTION                          ZP CONVENTION
  ════════════════                          ══════════════

  Caller prepares args:                     Caller prepares args:

    evaluate arg1 → AX                       evaluate arg1 → AX
    PHW .AX            ─┐ push to stack      STA $03; STX $04  ─┐ store to ZP
    evaluate arg2 → AX  │                    evaluate arg2 → AX  │
    PHW .AX            ─┘                    STA $05; STX $06  ─┘
    JSR _func                                JSR _func


  Callee reads params:                      Callee reads params:

    TSX                 ─┐ 4+ bytes          LDA $03           ─┐ 2 bytes
    LDA $0105,X         ─┘ per access        LDX $04           ─┘ per access


  Callee returns:                           Callee returns:

    RTS #4             (remove 4 param bytes) RTS              (nothing to clean up)


  Caller receives result:                   Caller receives result:

    ; result in AX (int)                     ; result in AX (int)
    ; or hidden-pointer (long/struct)        ; or AXYZ (long), hidden-ptr (struct)
```
